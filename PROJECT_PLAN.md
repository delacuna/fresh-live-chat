# Fresh Live Chat — ゲーム配信向けネタバレフィルター

## プロジェクト概要

ゲーム配信の視聴者・配信者向けのチャットフィルタリングサービス。ゲームの進行状況に連動し、その時点でネタバレとなるチャットメッセージを検出・非表示にする。

**2つのモード:**
- **アーカイブモード（MVP）**: 過去のアーカイブ配信のチャットリプレイからネタバレコメントをフィルタ
- **ライブモード（Phase 2以降）**: ライブ配信中のチャットからネタバレをリアルタイムでフィルタ

### 解決する課題

- 現状のNGワード機能ではネタバレを防げない（文脈依存のため）
- モデレーターが手動で削除しても、配信者の目に入った時点でネタバレは成立
- 削除されたメッセージがハイライト表示され、かえって目立つ問題
- 視聴者同士のネタバレも、アーカイブ視聴時の体験を損なう
- **アーカイブ特有の問題**: ユーザーをブロックしてもそのアーカイブ内ではコメントが表示され続ける
- **アーカイブ特有の問題**: 配信後にモデレーションされたコメントもチャットリプレイには残る

### ターゲットユーザー

- アーカイブ配信をネタバレなしで楽しみたい視聴者（MVP最優先）
- ストーリー重視のゲームを初見プレイする配信者（ライブモード）
- 配信者のモデレーター

### 競合状況（2026年4月時点）

- **直接競合: なし** — ゲーム進行連動型のネタバレフィルタは存在しない
- 既存のチャットBot（Nightbot, Streamlabs, Fossabot等）はキーワードベースのみ
- 既存のスポイラー防止拡張（Timeless, Anticipation等）はVODの再生バー非表示のみ
- Twitch公式にはスポイラー機能の要望が2016年から存在するが未実装
- 学術分野ではSpoilerNet（UC San Diego）が書評ネタバレ検出で89-92%の精度を実証

### アーカイブモード vs ライブモードの違い

| 項目 | アーカイブモード（MVP） | ライブモード（Phase 2） |
|------|----------------------|----------------------|
| 対象 | 過去の配信アーカイブ | ライブ配信中 |
| 利用者 | 視聴者 | 配信者 + 視聴者 |
| レイテンシ要件 | 緩い（先読み・バッチ処理可能） | 厳しい（<300ms） |
| 進行状況の設定 | 視聴者が手動設定 | 配信者が手動設定 |
| チャット取得方法 | DOM監視（MutationObserver） | DOM監視 + YouTube Live Chat API |
| 判定結果キャッシュ | 可能（同じ動画なら再利用） | 困難（リアルタイム） |
| 実装難易度 | 低〜中 | 中〜高 |

### コンポーネント詳細

#### 1. フロントエンド — Chrome Extension (YouTube)

**Chrome Extension（Manifest V3）**

アーカイブモード（MVP）:
- MutationObserverでチャットリプレイのDOM要素を監視
- 新しいコメント要素が追加されるたびにフィルタ判定を実行
- ネタバレと判定されたコメントをDOMから非表示（display:none またはカスタムUI で折りたたみ）
- 判定結果をローカルにキャッシュ（動画ID + コメントID → 判定結果）
- 同じ動画を2回目以降に見る場合はキャッシュからフィルタ適用（API呼び出し不要）
- 視聴者がポップアップUIからゲーム・進行状況を手動設定

ライブモード（Phase 2）:
- ライブ配信ページでも同様にMutationObserverでチャットを監視
- レイテンシ要件が厳しいため、Stage 1 の処理をService Worker内で高速実行
- 配信者が進行状況を設定

共通:
- ポップアップUI: ゲーム選択、進行状況設定（チャプター選択 or イベントチェック）、フィルタ強度調整
- URLを見てアーカイブ/ライブを自動判別し、適切なモードを起動
- Edge（Chromiumベース）でもそのまま動作、Firefox対応も容易

**Twitch Extension（将来対応）**
- Twitch Extensions SDK を使用
- Twitch内でネイティブに動作する専用UI

**技術スタック:**
- React + TypeScript
- Chrome Extension Manifest V3（YouTube MVP）
- Twitch Extensions SDK（将来）

**既知の制約 — OBSチャットについて:**
- OBSのチャットドック機能は、OBS内蔵のブラウザエンジンで動作しており、Chrome拡張の影響を受けない
- そのため、配信者がOBSのチャットドックを見ている場合、フィルタされていない生のチャットが表示される
- 配信者を守るには: (a) Chromeブラウザ上でYouTubeのチャットを開いて見る、または (b) 将来的にフィルタ済みチャットを表示するカスタムWebページをOBSブラウザソースとして読み込む方式を検討

**ネタバレコメント投稿者への通知方針:**
- フィルタされたことを投稿者に通知しない（シャドウミュート方式）
- 理由: 悪意あるユーザーの回避行動を防止、無自覚なユーザーへの不快感を避ける、YouTube APIの制約
- 投稿者の画面ではコメントは表示されたまま、拡張導入者の画面からのみ非表示になる

**インストール範囲と設定共有:**
- Chrome拡張はブラウザごとにローカルに動作するため、配信者と視聴者はそれぞれ個別にインストールが必要
- 配信者がフィルタをオンにしても、視聴者のブラウザには影響しない
- メリット: 視聴者が自分の進行状況に合わせて個別にフィルタレベルを設定可能
- 将来対応（Phase 4）: 配信者が「ゲーム名 + 現在の進行状況」を共有コード/URLとして公開し、視聴者の拡張がワンクリックで同じ設定を適用できる機能を検討

**フィルタのオンオフ・モード切替:**
- ポップアップUIにトグルスイッチ（ワンクリックでオン/オフ）
- キーボードショートカット対応（例: Alt+S）で配信中でも即座に切替可能
- 3段階のフィルタモード:
  - **厳格モード**: 明示的ネタバレ + 伏線の匂わせ + 攻略ヒントをすべてブロック
  - **標準モード（デフォルト）**: 明示的ネタバレ + 伏線の匂わせをブロック、攻略ヒントは許可
  - **緩めモード**: 明示的ネタバレのみブロック、匂わせ・攻略ヒントは許可
  - **オフ**: フィルタ無効（配信者がアドバイスを求めたい場合など）
- Stage 2 の判定カテゴリ（direct_spoiler / foreshadowing_hint / gameplay_hint / safe）と連動

**フィルタされたメッセージの表示方式:**

ユーザーが以下の2つから選択可能（ポップアップUIで設定）:

- **プレースホルダー表示（デフォルト）**: コメントの内容を「⚠ ネタバレの可能性があるためフィルタされました（クリックで表示）」に置換。チャットの流れを自然に保ちつつ、フィルタが機能していることを可視化。クリックすると元のコメントを確認でき、誤判定時の自己修正にも対応
- **完全非表示**: コメントをDOMから完全に非表示（display:none）。最もクリーンな体験だが、チャットの密度が不自然に薄く見える場合がある

**フィルタカウンター（常時表示）:**
- チャット欄の上部またはポップアップUIに「🛡 N件のコメントをフィルタしました」と表示
- セッション（動画単位）ごとにカウント
- アドブロックの「○件の広告をブロック」と同様の心理効果で、拡張の価値を実感しやすくする
- 技術的にはContent Script内の変数インクリメント + DOM更新のみ、負荷はゼロ

**パフォーマンス影響（PCへの負荷）:**

| 処理 | 実行場所 | CPU影響 | メモリ影響 |
|------|---------|---------|-----------|
| MutationObserver（DOM監視） | ブラウザ（Content Script） | 無視できるレベル | ~1MB以下 |
| Stage 1 キーワードマッチ | ブラウザ（Service Worker） | マイクロ秒/メッセージ | 数MB（キーワードリスト） |
| Stage 2 LLM判定 | バックエンドサーバー | ゼロ（HTTP通信のみ） | ゼロ |
| 判定結果ローカルキャッシュ | ブラウザ（chrome.storage） | 無視できるレベル | 数MB |

- 合計メモリ使用量: ~10MB以下（一般的なChrome拡張と同程度）
- CPUへの影響: OBSエンコードやゲーム処理と比べて誤差の範囲
- ネットワーク: Stage 2 に到達するメッセージのみAPI呼び出し（全体の~10%）

#### 2. バックエンド — 判定エンジン

**2段階フィルタリングアーキテクチャ:**

```
チャットメッセージ受信
        │
        ▼
  ┌─────────────┐     高速（<10ms）
  │ Stage 1:    │     キーワード / 埋め込みベクトル類似度
  │ 粗フィルタ   │     → 明らかに安全なメッセージをパス
  └──────┬──────┘
         │ 疑わしいメッセージのみ
         ▼
  ┌─────────────┐     中速（50-200ms）
  │ Stage 2:    │     LLM（軽量モデル）による文脈判定
  │ 精密判定    │     → ネタバレ度合いをスコアリング
  └──────┬──────┘
         │
         ▼
    判定結果を返却
    (PASS / HOLD / BLOCK)
```

**技術スタック候補:**
- Node.js (TypeScript) または Python (FastAPI)
- Redis: チャットメッセージのキューイング、キャッシュ
- PostgreSQL: ゲームデータ、ユーザー設定の永続化
- WebSocket: リアルタイム通信

`apps/api/src/services/knowledge/base.ts` に抽象化層を設け、裏側のストレージを切り替え可能にしておく。

#### 4. 判定ロジック

**Stage 1 — 粗フィルタ（高速）:**
- 現在の進行状況（チャプター or 通過済みイベント）以降の `spoiler_entities` に含まれるキーワードとの照合
- テキスト埋め込みベクトルの類似度計算（セマンティック検索）
- 閾値以下 → PASS（安全）、以上 → Stage 2へ

**Stage 2 — LLM判定（精密）:**
- プロンプト例:
  ```
  あなたはゲームのライブ配信チャットのネタバレ判定AIです。
  ゲーム: {game_title}
  現在の進行状況: {current_progress}
  以下のチャットメッセージを判定してください。

  メッセージ: "{chat_message}"

  判定基準（spoiler_category）:

  "direct_spoiler" — 明示的なネタバレ（重度）
    現在の進行状況より先のストーリー展開、キャラクターの生死、
    真相、結末などを直接的に述べている。
    例: 「○○は実は裏切り者だよ」「ラスボスは○○」

  "foreshadowing_hint" — 伏線の指摘・匂わせ（中度）
    先の展開を知っている人が、初見を装いつつ特定の場面・台詞・
    キャラクターに注意を向けさせるコメント。
    明示的なネタバレワードを含まなくても、初見プレイヤーが
    自然に気づくはずの伏線や発見を先回りして示唆している。
    例: 「ここ覚えておいて」「今の会話重要だよ」「この人怪しいな...（意味深）」
        「あっ...」「初見だけどこれ絶対あとで関係あるよね」
        「このアイテム捨てない方がいいよ」
    判定のポイント: そのコメントを読んだ初見視聴者が、
    本来なら気に留めなかったはずのことに注意を向けてしまうかどうか。

  "gameplay_hint" — 攻略ヒント（軽度）
    次に何をすべきか、どこに行くべきかなどのゲームプレイに関する指示。
    ストーリーには直接触れないが、自力での探索・発見を損なう。
    例: 「左の道に行った方がいいよ」「そのボスは炎属性が弱点」

  "safe" — 安全
    既に通過した内容への言及、ゲームと無関係な会話、
    純粋な感想（「このBGM最高」「グラフィック綺麗」）、
    配信者への応援（「がんばれ」「草」）。

  JSON形式で回答:
  {
    "spoiler_category": "direct_spoiler" | "foreshadowing_hint" | "gameplay_hint" | "safe",
    "confidence": 0.0-1.0,
    "reason": "判定理由を簡潔に"
  }
  ```

**フィルタモードと判定カテゴリの連動:**

| フィルタモード | direct_spoiler | foreshadowing_hint | gameplay_hint | safe |
|-------------|---------------|-------------------|--------------|------|
| 厳格 | ブロック | ブロック | ブロック | 表示 |
| 標準 | ブロック | ブロック | 表示 | 表示 |
| 緩め | ブロック | 表示 | 表示 | 表示 |

※ デフォルトは「標準」モード（匂わせ・伏線指摘もブロック、攻略ヒントは許可）

- 使用モデル候補: Claude Haiku（高速・低コスト）、またはファインチューニング済み軽量モデル

**匂わせ判定の精度向上戦略:**
- まずはプロンプト設計のみで開始し、実際のコメントでテスト
- 誤判定パターンを収集し、プロンプトを継続改善
- 将来的に精度が不足する場合: 誤判定報告データからfew-shot examplesをプロンプトに追加
- 知識ベースに「この場面には重要な伏線がある」フラグを追加し、特定シーン前後で匂わせ判定の感度を上げることも可能

## MVP（Minimum Viable Product）定義

### MVP スコープ — アーカイブモード

**含めるもの:**
- Chrome Extension for YouTube（アーカイブのチャットリプレイフィルタ）
- ポップアップUI: ゲーム選択、進行状況の手動設定
- 対応ゲーム: 3-5本の人気ストーリーゲーム（チャプター型・イベント型の両方を含む）
- 手動での進行状況設定（チャプター選択 or イベントチェックリスト）
- 2段階フィルタリング（キーワード + LLM）
- ネタバレメッセージの非表示（視聴者のチャットリプレイ表示から除外）
- 判定結果のローカルキャッシュ（同じ動画の再視聴時に即座にフィルタ適用）

## 技術的課題と対策

### 1. レイテンシ

| 要件 | 目標値 |
|------|--------|
| Stage 1 判定 | < 10ms |
| Stage 2 判定 | < 200ms |
| エンドツーエンド | < 300ms |

**対策:**
- Stage 1で90%以上のメッセージをPASSし、LLM呼び出しを最小化
- LLMレスポンスのキャッシュ（類似メッセージへの再利用）
- エッジロケーションでのStage 1処理

### 2. 精度

| 指標 | 目標値 |
|------|--------|
| Recall（ネタバレ検出率） | > 85% |
| Precision（誤検出率の低さ） | > 90% |

**対策:**
- 誤判定時は「安全側」に倒す（疑わしければブロック）
- 配信者が判定レベルを調整可能（厳格 / 標準 / 緩め）
- 誤判定の報告からモデルを継続改善

### 3. コスト

**LLM API呼び出しコストの見積もり:**
- 平均チャット速度: 30メッセージ/分（アクティブな配信）
- Stage 1通過率: 10%（3メッセージ/分がStage 2へ）
- 1配信（4時間）あたり: 約720 LLM呼び出し
- Claude Haiku想定: ≈ $0.05-0.10 / 配信

**対策:**
- Stage 1の精度向上でLLM呼び出しを削減
- バッチ処理（数メッセージをまとめて判定）
- キャッシュの活用

### 4. 回避行動への対策

ユーザーがフィルタを回避しようとするパターン:
- 伏字（「○○が死ぬ」→「○が○ぬ」）
- 暗喩（「あの人、もういないよ」）
- 絵文字による表現
- ローマ字 / 英語混じり

**対策:**
- LLM（Stage 2）は文脈理解ができるため、多くの回避を検出可能
- 既知の回避パターンをStage 1に追加
- 完全な防止は不可能だが、カジュアルなネタバレの大半を防げればOK

### 5. 多言語対応

VTuberなど国際的な視聴者が多い配信では、英語・韓国語・スペイン語等でもネタバレコメントが飛ぶ。

**ネタバレ判定の多言語対応（重要度: 高）:**
- Stage 2（LLM）: Claude Haikuは多言語を理解するため、知識ベースが日本語でも他言語のネタバレを判定可能。追加実装は不要
- Stage 1（キーワード）: `aliases` フィールドで多言語の表記揺れに対応（例: "ゴドリック" / "Godrick" / "고드릭"）
- MVP段階では日本語 + 英語のエイリアスを登録、他の言語は需要に応じて追加

**UIの多言語対応（i18n）（重要度: 低 — 将来対応）:**
- MVPでは日本語UIのみ
- Chrome拡張のi18n機能（`_locales/` ディレクトリ）がManifest V3に標準で組み込まれているため、後から追加が容易
- ユーザー数が増え、国際展開する段階で対応すれば十分

### 6. 他のChrome拡張との互換性（Flow Chat問題）

**問題:**
Flow Chat for YouTube Live（約10万ユーザー）などの弾幕系拡張は、チャットリプレイのDOM要素からテキストを読み取り、動画上に流す。Fresh Live Chatが `display:none` でコメントを非表示にしても、Flow Chatがテキストを先に読み取っていれば画面に流れてしまう。

**解決策 — テキスト書き換え方式:**
`display:none` ではなく、コメント要素のテキスト内容自体を書き換える。

```
フィルタ前: <span id="message">ラスボスは○○だよ</span>
フィルタ後: <span id="message" data-fresh-live-chat-original="ラスボスは○○だよ"
                 data-fresh-live-chat-filtered="true">
              ⚠ フィルタされました
            </span>
```

- 元テキストは `data-fresh-live-chat-original` 属性に退避（「クリックして表示」で復元可能）
- Flow Chatが読み取る時点でテキストが書き換わっているため、弾幕にもフィルタ済みテキストが流れる
- 完全非表示モードの場合は、テキスト書き換え + `display:none` の両方を適用
- この方式はFlow Chat以外の類似拡張（Bulletchat for YouTube等）にも有効

### 運営コスト

| 項目 | MVP段階 | ユーザー100人 | ユーザー1,000人 |

### マネタイズ方針

| 段階 | 方針 | 理由 |
|------|------|------|
| 初期 | 完全無料 | ユーザー獲得とフィードバック収集が最優先 |

後から「誰がどれだけ使っているか」が把握できないと、有料化の判断ができない。

## セキュリティ設計

### 不要なもの

- ユーザーアカウント登録・パスワード管理
- OAuth連携 / WebAuthn
- メールアドレスの収集
- 個人情報の保存

ユーザーはChrome拡張をインストールするだけで使い始められる。アカウント作成は一切不要。

### 必要なもの — APIプロキシの保護

Chrome拡張のコードは誰でも閲覧できるため、APIプロキシのURLは公開される前提で設計する。

**3つの保護レイヤー:**

1. **匿名トークン**: 拡張インストール時にランダムUUIDを生成し、chrome.storageに保存。APIリクエストのヘッダーに付与。プロキシ側で「拡張から来たリクエストか」を最低限検証する
2. **レート制限**: トークン単位 + IPアドレス単位で、1分あたり・1日あたりのリクエスト数を制限。大半の悪用を防止
3. **月間利用上限**: トークンごとにAPI呼び出し回数を追跡。上限超過時は制限。将来の有料化にもそのまま使える

**ユーザーのプライバシー:**
- チャットメッセージはネタバレ判定のためにAPIプロキシに送信されるが、サーバー側でログに保存しない
- 判定結果はchrome.storageにローカル保存（サーバーに蓄積しない）
- プライバシーポリシーでデータの取り扱いを明示（Chrome Web Store審査でも必要）

## 将来構想: カスタムフィルタ機能

ネタバレだけでなく、ユーザーが任意のコメントカテゴリをフィルタできる機能。

### フィルタカテゴリ（想定）

| カテゴリ | 説明 | ゲーム知識ベース必要 | 実装段階 |
|---------|------|-------------------|---------|
| 明示的ネタバレ | ゲームの先の展開を直接述べる | 必要 | MVP |
| 伏線の匂わせ | 先を知る人が初見を装い注意を向けさせる | 必要 | MVP |
| 攻略ヒント | 次にやるべきことの指示 | 必要 | MVP |
| 暴言・罵倒 | 悪意のある攻撃的な言葉 | 不要 | 将来 |
| 配信者への誹謗中傷 | 配信者個人への攻撃 | 不要 | 将来 |
| 無関係な配信者の名前 | 他チャンネルの宣伝・比較 | 不要 | 将来 |
| 他コメントへの攻撃的な返信 | 視聴者間の口論 | 不要 | 将来 |

### 技術的な実現方法

現在の2段階フィルタ構造をそのまま流用可能。Stage 2のLLMプロンプトをマルチラベル分類に拡張:

```
このコメントを以下のカテゴリに分類してください（複数可）:
- spoiler: ゲームのネタバレ
- hint: 攻略ヒント
- abuse: 暴言・罵倒
- harassment: 配信者への誹謗中傷
- off_topic_creator: 無関係な配信者への言及
- toxic_reply: 攻撃的な返信
- safe: 問題なし

JSON形式で回答: {"categories": [...], "confidence": 0-1}
```

一回のAPI呼び出しで複数カテゴリを同時に判定できるため、追加コストはほぼゼロ。

### 戦略的な意味

ゲーム知識ベースが不要なカテゴリ（暴言、誹謗中傷等）は、ゲームを選択していない状態でも単独で機能する。
つまり「ネタバレフィルタを使わなくても、コメントフィルタとしてだけ入れる価値がある」拡張になり、
ユーザーベースをゲーム配信の視聴者以外にも広げられる可能性がある。

### 2. 次のステップ: Phase 0 に移行

```bash
mkdir fresh-live-chat && cd fresh-live-chat
git init
```

Claude Code に伝える:

> 「Fresh Live Chatというゲーム配信向けネタバレフィルタのプロジェクトを始めます。
> PROJECT_PLAN.mdにアーキテクチャと計画があります。
> Phase -1のプロトタイプで、YouTubeのチャットリプレイの
> DOM構造とMutationObserverでの監視が動くことを確認済みです。
> 以下の知見が得られています:
> - チャットリプレイはiframe内にあり、all_frames: true が必須
> - #items 要素は遅延初期化されるため、MutationObserverで出現を検知する必要がある
> - display:none だけではFlow Chat等の他拡張に効かない（テキスト書き換え方式が必要）
>
> Phase 0に進みます。モノレポ構成を作り、packages/sharedの型定義から始めてください。」

### 3. 一人開発の心得

- **動く状態を常に維持**: Gitコミットは細かく、1機能ずつ
- **設計書は仮説**: 実装しながら発見したことで柔軟に変更してよい
- **3日間何も動かない状態は危険信号**: スコープを縮小するか、別のタスクに切り替える
- **トラブルはTROUBLESHOOTING.mdに記録**: 同じ問題の再発防止と学習の蓄積
